مسئله اصلی این است که خروجی فعلی فقط داخل قاب 9:16 نمایش داده می‌شود، اما خود تصویر تولیدشده الزاماً 9:16 واقعی نیست؛ به همین خاطر بالا و پایین خالی/مشکی دیده می‌شود. همچنین لاگ backend نشان می‌دهد نسخه‌ای از تابع هنوز با خطای `Unsupported image type` درگیر بوده، پس باید مسیر backend را تثبیت کنیم نه فقط UI را عوض کنیم.

Plan:

1. اصلاح backend تابع `image-reframe`
   - ورودی اصلی را همان تصویر کاربر نگه می‌داریم.
   - برای هر نسبت هدف، یک «تصویر مرجع ابعاد» کوچک و واقعی با همان نسبت می‌سازیم و آن را به عنوان آخرین تصویر به Nano Banana می‌دهیم.
   - در prompt صریح می‌کنیم که تصویر اول منبع محتواست و تصویر دوم فقط مرجع نسبت/ابعاد است، نه محتوای بصری.
   - هدف: مدل واقعاً تصویر را outpaint/reframe کند، نه اینکه خروجی قبلی را وسط یک قاب بگذارد.

2. اعتبارسنجی خروجی قبل از ذخیره
   - بعد از دریافت تصویر از Nano Banana، ابعاد واقعی فایل خروجی را از bytes می‌خوانیم.
   - اگر نسبت خروجی با هدف یکی نبود، یک retry کنترل‌شده با prompt سخت‌گیرانه‌تر انجام می‌دهیم.
   - اگر باز هم درست نبود، خطای واضح برمی‌گردانیم تا خروجی اشتباه با فضای خالی به کاربر نمایش داده نشود.

3. پاک‌سازی منطق قبلی و جلوگیری از خطای bundle/unsupported image
   - هیچ import خارجی مثل `imagescript` استفاده نمی‌کنیم.
   - هیچ decode سنگین WebP/compose محلی انجام نمی‌دهیم.
   - فقط از fetch، base64، و parser سبک ابعاد PNG/JPEG/WebP استفاده می‌کنیم تا deploy پایدار بماند.

4. اصلاح frontend برای نمایش نتیجه واقعی
   - بعد از پاسخ backend، اگر backend هشدار یا خطای نسبت تصویر داد، toast مناسب نمایش داده می‌شود.
   - preview همچنان با `object-contain` باقی می‌ماند، اما چون فایل خروجی واقعاً نسبت درست دارد دیگر فضای خالی مصنوعی نباید دیده شود.

5. اعتبارسنجی نهایی
   - تابع `image-reframe` deploy می‌شود.
   - با فراخوانی مستقیم backend تست می‌کنم که تابع اجرا شود و دیگر خطای bundle/import یا `Unsupported image type` ندهد.
   - نتیجه را فقط بعد از تست موفق اعلام می‌کنم.
