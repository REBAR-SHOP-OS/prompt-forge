## Root cause

When the Scenario Writer 45s flow chains Scene 1 → 2 → 3, each scene is queued as a 15s Veo (`flow-video-1` → `veo-3.1-generate-preview`) job. Veo only renders 8s base clips, so the adapter chains an `extend` call to reach 15s.

Edge logs show Scene 1 failed at `2026-05-19T17:03:34Z` with:

```
veo extend failed 400 "Input video must be a video that was generated by VEO that has been processed."
```

The phase-1 operation returns `done:true` with a `video.uri`, but Google Veo's `predictLongRunning` extend endpoint rejects that URI for a short window after generation while the file is still being "processed" server-side. Today we treat this 400 as a hard failure → the whole scenario chain aborts at Scene 1 with the "Scene 1 failed; cannot chain remaining scenes" message the user sees.

## Fix (single file, backend only)

Edit `supabase/functions/_shared/modules/external-api-adapter/service.ts`:

1. **Extend `VeoState`** with two optional fields:
   - `pendingExtensionUri?: string` — the phase-1 video URI we still need to extend.
   - `extensionAttempts?: number` — retry counter for the extend call.

2. **In `pollVeo`, the `hasExtension && !state.extensionStarted` branch** (around lines 720–751): catch the `startVeoExtension` error. If the message matches a new helper `isVeoNotProcessedYet(msg)` (matches `"must be a video that was generated by VEO that has been processed"` and similar "not processed"/"still processing" variants), do NOT mark the job failed. Instead:
   - Persist `pendingExtensionUri = uri`, increment `extensionAttempts`, keep `extensionStarted: false`.
   - Return `status: "processing"` with the updated `providerJobId` (re-encoded state) and a modest `progressPercent` (~50).
   - Cap retries at ~6 (≈30–60s of polling backoff) before falling through to the existing hard-fail path so we don't loop forever if Google really rejected the clip.

3. **On the next poll**, detect `state.pendingExtensionUri` set + `extensionStarted: false` and re-try `startVeoExtension(state, state.pendingExtensionUri, apiKey)` directly without re-polling phase 1 (phase 1 is already done). On success, transition into the existing `extensionStarted: true` state with the new `currentOp`.

4. **Also treat the same "not processed" message as transient** inside `isTransientVeoError` so any other Veo call hitting it during phase-1 polling is retried instead of killing the job.

No changes to:
- `submitScenesAsJobs` / `DashboardPage.tsx` — the scenario UI flow is already correct.
- `jobs-create` / orchestrator gateway — they already persist evolved `providerJobId` after each poll.
- Veo state encoding format prefix (`veo:v1:`) — new fields are optional and backward-compatible with existing in-flight jobs.
